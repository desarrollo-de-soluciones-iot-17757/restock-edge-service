# Restock Edge Service

`restock-edge-service` is the IoT edge API responsible for receiving, validating and storing telemetry generated by Restock smart inventory devices.

This service acts as an intermediate layer between physical IoT devices and the Restock platform, allowing device-originated data to be processed closer to the source before being synchronized or integrated with cloud services.

The project follows a Domain-Driven Design (DDD) approach, separating device authentication, telemetry ingestion and persistence concerns into clearly defined layers and bounded contexts.

## Purpose

The main purpose of the Edge Service is to support unattended communication between Restock smart devices and the inventory management ecosystem.

In the current version, the service focuses on:

- authenticating IoT devices through a device identifier and API key
- receiving weight telemetry from smart scale devices
- validating telemetry payloads before persistence
- normalizing timestamps to UTC
- storing telemetry records using a lightweight persistence layer
- organizing the codebase through DDD-inspired bounded contexts

## Domain-Driven Design Approach

The Edge Service is organized around business capabilities instead of purely technical folders. This improves maintainability and allows each part of the system to evolve independently.

The current implementation is divided into two main bounded contexts.

## Bounded Contexts

### IAM

The IAM bounded context is responsible for identifying devices and validating their credentials before allowing telemetry ingestion.

Main responsibility:

- authenticate device-originated requests

Main domain concept:

- `Device`

### Devices

The Devices bounded context is responsible for managing the thresholds assigned to a specific device for validation after telemetry ingestion.

Main responsibility:

- manage device-specific thresholds for telemetry validation

Main domain concept:

- `DeviceThreshold`

### Tracking

The Tracking bounded context is responsible for receiving, validating and storing telemetry generated by IoT devices.

Main responsibility:

- register weight measurements from smart inventory devices

Main domain concept:

- `WeightRecord`

## Architecture

Each bounded context follows a layered structure inspired by DDD principles.

### Domain Layer

Contains the core business concepts and rules of the bounded context.

Responsibilities:

- define entities
- enforce domain rules
- keep business logic independent from frameworks and persistence tools

### Application Layer

Coordinates use cases and application-specific workflows.

Responsibilities:

- orchestrate domain services and repositories
- handle application-level operations
- expose use cases to the interface layer

### Infrastructure Layer

Contains technical implementations required by the application.

Responsibilities:

- define ORM models
- implement repositories
- manage persistence details
- interact with external infrastructure components

### Interfaces Layer

Exposes the service capabilities through HTTP endpoints.

Responsibilities:

- receive HTTP requests
- validate request structure
- invoke application services
- return API responses

## API Overview

### Register Weight Telemetry

```http
POST /api/v1/tracking/weight-records
```

Registers a new weight measurement generated by an authenticated Restock smart scale device.

### Required Headers

```http
Content-Type: application/json
X-API-Key: <device-api-key>
```

### Request Body

```json
{
  "device_id": "device-identifier",
  "weight": 1200.5,
  "created_at": "2026-05-25T12:00:00-05:00"
}
```

### Request Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `device_id` | string | Yes | Unique identifier of the IoT device |
| `weight` | number | Yes | Weight measurement reported by the device |
| `created_at` | string | No | ISO 8601 timestamp of the measurement |

When `created_at` is provided, the service normalizes the timestamp to UTC before storing it.

### Success Response

```http
201 Created
```

```json
{
  "id": 1,
  "device_id": "device-identifier",
  "weight": 1200.5,
  "created_at": "2026-05-25T17:00:00+00:00"
}
```

### Error Responses

```http
400 Bad Request
```

Returned when the request payload is incomplete or invalid.

Common causes:

- missing required fields
- invalid weight value
- malformed timestamp

```http
401 Unauthorized
```

Returned when the device cannot be authenticated.

Common causes:

- missing API key
- missing device identifier
- invalid device credentials
